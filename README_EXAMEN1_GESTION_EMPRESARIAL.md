# Examen — Sistemas de Gestión Empresarial
### Proyecto: Ratatui — Recetario Inteligente con IA

---

## 1. Ollama + Modelos de IA Local

Ratatui es un recetario inteligente que funciona íntegramente con inteligencia artificial local gracias a Ollama. La idea es simple: en lugar de depender de APIs de pago externas, toda la inteligencia del sistema corre en tu propia máquina, sin enviar ningún dato al exterior. El programa usa dos modelos distintos según la tarea: uno especializado en entender el significado del texto y otro en generar respuestas conversacionales, lo que permite una experiencia completa y autónoma.

Para usarlo, primero hay que tener Ollama instalado y descargar los dos modelos con `ollama pull bge-m3:latest` y `ollama pull qwen2.5:7b`. Una vez hecho esto, se lanza el backend con `uvicorn main:app --reload` y se abre el navegador en `http://localhost:8000`. A partir de ahí el sistema ya está operativo: el modelo de embeddings trabaja en silencio cada vez que buscas o guardas una receta, y el modelo de chat responde cuando interactúas con el asistente.

Desde el punto de vista del usuario, lo más visible es el asistente "Chef Ratatui": puedes preguntarle cosas como "¿qué hago con pollo y limón?" y él buscará en tu recetario, pensará qué recetas encajan y te las presentará con detalle. Todo esto ocurre localmente, sin latencias de red ni costes por token. La búsqueda de recetas en la barra lateral también usa embeddings vectoriales para encontrar coincidencias semánticas, no solo por palabras exactas.

Para los embeddings se usa **bge-m3:latest**, un modelo multilingüe de BGE (Beijing Academy of AI) optimizado para recuperación de información. Es una elección muy justificada: soporta español sin degradación de calidad, genera vectores de 1024 dimensiones con alta capacidad discriminativa, y en local corre razonablemente rápido (entre 50 y 200 ms por embedding) sin consumir demasiada VRAM. Comparado con modelos como `nomic-embed-text`, bge-m3 puntúa significativamente mejor en benchmarks multilingües como MIRACL.

Para la generación de texto se usa **qwen2.5:7b**, el modelo de lenguaje de Alibaba de 7.000 millones de parámetros. La razón de esta elección es el equilibrio: modelos más pequeños como phi-3.8b dan respuestas menos coherentes en español, y modelos más grandes como llama3:70b requieren hardware que la mayoría no tiene. Qwen2.5:7b funciona con 8GB de RAM, genera texto en streaming a unos 20-30 tokens por segundo en un Mac moderno, y tiene un buen comportamiento siguiendo instrucciones complejas en español, que es exactamente lo que necesita el asistente culinario.

## 2. Entrenamiento de IA con Ollama Modelfile

Este ejercicio demuestra cómo personalizar un modelo de IA base para que adopte una identidad y comportamiento específicos, lo que en el mundo real se conoce como fine-tuning o ajuste fino. En lugar de partir siempre de cero, Ollama permite crear un modelo derivado que ya lleva "grabadas" ciertas instrucciones de comportamiento, reduciendo el trabajo del sistema en tiempo de ejecución y haciendo que las respuestas sean más consistentes. El proyecto incluye un endpoint que genera este fichero de configuración de forma dinámica.

El proceso es sencillo: se llama al endpoint `GET /api/modelfile` y la API devuelve un JSON con el contenido de un archivo `Modelfile` y las instrucciones de uso. Ese contenido se guarda en un fichero llamado `Modelfile` y luego se ejecuta `ollama create ratatui-chef -f Modelfile`. Esto crea un nuevo modelo llamado `ratatui-chef` que es una copia de qwen2.5:7b pero con el sistema de instrucciones de Ratatui ya incorporado. Para ver el diferencial basta con comparar: primero ejecutas `ollama run qwen2.5:7b` y preguntas "¿qué hago si tengo pollo?", y luego haces lo mismo con `ollama run ratatui-chef`. El modelo base responde de forma genérica; el modelo entrenado responde como un chef con acceso a un recetario.

Una vez creado el modelo personalizado, se puede actualizar la variable `LLM_MODEL` en `main.py` para que toda la aplicación use `ratatui-chef` en lugar del modelo base. El efecto es inmediato: las respuestas del chat son más coherentes con el rol culinario, el modelo ya no necesita que le recuerden quién es en cada conversación, y los tiempos de respuesta mejoran ligeramente porque el contexto del sistema prompt se procesa con prioridad. También se puede comparar la calidad de las respuestas antes y después del entrenamiento haciendo las mismas preguntas a ambos modelos y anotando las diferencias.

La tecnología que hace esto posible es el sistema de **Modelfiles** de Ollama, que sigue una sintaxis inspirada en los Dockerfiles. Las directivas clave son `FROM` (qué modelo base usar), `SYSTEM` (el prompt de sistema que define el rol), y `PARAMETER` (ajustes como temperatura, top_p y contexto). El system prompt de Ratatui define cuándo usar herramientas, cómo citar recetas, en qué idioma responder y con qué tono, todo ello empaquetado en el modelo para que sea persistente entre sesiones.

El diferencial entre antes y después del entrenamiento es medible y observable. Sin el Modelfile personalizado, el modelo responde en inglés con frecuencia, no entiende que debe buscar en un recetario, y da respuestas de cocina genéricas. Con el modelo `ratatui-chef`, responde siempre en español, sabe cuándo llamar a las herramientas de búsqueda, y tiene la personalidad cálida y experta del chef. Además, los parámetros de `temperature: 0.4` y `top_p: 0.95` ajustados en el Modelfile hacen que las respuestas sean más precisas y menos "alucinadoras" que con los valores por defecto del modelo base.

## 3. MCP — Model Context Protocol (MiniSaaS)

El MCP (Model Context Protocol) es el estándar que define cómo una IA puede conectarse de forma estructurada con aplicaciones y herramientas externas. En Ratatui hemos implementado un servidor MCP completo que permite al asistente acceder al recetario como si fuera una base de datos: el usuario escribe en lenguaje humano, la IA transforma esa petición en una llamada a herramienta estructurada, el sistema la ejecuta, y la IA interpreta los resultados y los devuelve en un formato legible. Es el corazón de lo que hace que Ratatui sea un MiniSaaS real y no solo un chatbot.

El flujo de uso es el siguiente: el usuario abre el chat y escribe algo como "quiero hacer una cena italiana para cuatro personas que no lleve más de 40 minutos". El asistente recibe ese mensaje y, antes de responder, invoca automáticamente la herramienta `search_recipes` con los parámetros `{query: "italiana cena", max_time: 40, n: 3}`. El sistema ejecuta esa búsqueda vectorial en ChromaDB y devuelve los chunks de recetas más relevantes. La IA recibe esos datos crudos y los convierte en una respuesta natural, fluida y contextualizada para el usuario.

Las herramientas MCP expuestas también son accesibles directamente desde fuera de la aplicación, lo que convierte a Ratatui en un SaaS real. Cualquier cliente externo puede hacer un `GET /api/mcp/tools` para descubrir qué herramientas están disponibles y sus esquemas, y un `POST /api/mcp/call` con `{"tool": "search_recipes", "params": {"query": "pasta"}}` para invocarlas directamente. Esto permite que otros programas o agentes de IA se conecten a Ratatui como si fuera un plugin, siguiendo el protocolo MCP estándar de Anthropic.

Las tres herramientas MCP implementadas son `search_recipes` (búsqueda semántica con filtros de tiempo y dificultad), `list_recipes` (inventario completo del recetario) y `get_recipe` (detalle completo de una receta por ID). Cada herramienta está definida con su nombre, descripción y esquema de parámetros en formato JSON Schema, que es exactamente el formato que el LLM usa para decidir cuándo y cómo llamarlas. La IA ha sido entrenada para saber qué herramienta usar en cada situación sin que el usuario tenga que especificarlo.

La implementación técnica usa la API de herramientas (tool use) de Ollama, que permite pasar un array de definiciones de herramientas junto con el mensaje del usuario. Cuando el modelo detecta que necesita buscar en el recetario, devuelve una respuesta con `tool_calls` en lugar de texto, y el backend ejecuta la herramienta correspondiente en `execute_tool()`. El resultado se inyecta de vuelta en el historial de mensajes como un mensaje de rol `tool`, y el modelo continúa la conversación con esa información. Este ciclo puede repetirse hasta 4 veces en una sola interacción, permitiendo búsquedas complejas con múltiples pasos.

## 4. IA Agéntica — Ratatui como Agente Autónomo

Una IA agéntica es aquella que no se limita a responder preguntas puntuales, sino que recibe una misión y trabaja de forma autónoma para cumplirla, tomando decisiones por sí misma sobre qué pasos dar. Ratatui implementa este paradigma en su sistema de chat: cuando recibes un mensaje del usuario, el asistente no simplemente genera una respuesta; lanza un proceso independiente en background que razona, busca información, evalúa los resultados y construye la respuesta final, todo ello sin intervención humana. Si el cliente cierra la pestaña del navegador a mitad de una respuesta, el agente sigue trabajando y guarda el resultado cuando termina.

El uso desde el punto de vista del usuario es completamente transparente: simplemente escribe un mensaje en el chat y, mientras el agente trabaja, aparecen en tiempo real unos "pensamientos" que muestran qué está haciendo ("Buscando «pasta cremosa» en el recetario…", "Consultando detalles de la receta…", "Preparando la respuesta…"). Estos mensajes de pensamiento llegan vía SSE (Server-Sent Events) mientras el agente ejecuta su loop de razonamiento. Una vez que ha recopilado toda la información necesaria, el texto de la respuesta llega también en streaming, token a token, con las fuentes numeradas al final.

Lo que hace al agente verdaderamente autónomo es que decide por sí mismo cuántas búsquedas necesita hacer. Si la primera búsqueda devuelve resultados insuficientes, puede hacer una segunda con términos distintos. Si el usuario pide algo muy específico, puede combinar `search_recipes` para encontrar candidatas y luego `get_recipe` para obtener los detalles de la mejor opción. Todo esto sin que el usuario tenga que guiarle paso a paso. También hay un fallback automático: si el modelo no usa herramientas (cosa que puede pasar), el agente activa el pipeline RAG directamente como plan B.

La arquitectura técnica que hace posible este comportamiento se basa en `asyncio.Task` de Python, que permite lanzar la lógica del agente como una tarea completamente independiente del ciclo de vida de la petición HTTP. El diccionario `_active` rastrea las tareas en curso por `chat_id`, lo que permite que múltiples clientes se suscriban al mismo stream o que un cliente reconecte y retome donde lo dejó. La `asyncio.Queue` actúa como canal de comunicación entre la tarea del agente y el generador SSE que sirve los eventos al cliente.

El loop agéntico en sí está implementado en la función `_process_chat` y puede iterar hasta 4 veces sobre el ciclo razonamiento → llamada a herramienta → resultado. En cada iteración, el agente llama al LLM con el historial completo más las definiciones de herramientas MCP, analiza si la respuesta incluye `tool_calls`, y si es así ejecuta esas llamadas e incorpora los resultados al contexto antes de la siguiente iteración. Cuando el modelo devuelve una respuesta sin `tool_calls`, el loop termina y se pasa a la generación de la respuesta final en streaming. El historial de la conversación se persiste en ChromaDB al finalizar, incluyendo el título generado automáticamente por la IA.
